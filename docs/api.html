Overview 概览

Once deployed, HaeronClaw exposes two HTTP chat endpoints, an MCP server endpoint, and a built-in browser Chat UI. All endpoints accept standard HTTP requests with JSON bodies and GitHub OAuth tokens for authentication. 部署后，HaeronClaw 提供两个 HTTP 聊天端点、一个 MCP server 端点以及内置的浏览器聊天 UI。所有端点接受标准 HTTP 请求，使用 JSON body 和 GitHub OAuth token 认证。

Getting the Base URL 获取 Base URL

After deployment, get the container app hostname using the Azure CLI: 部署后，可以用 Azure CLI 获取 Container App 的 hostname：

bash 
APP_NAME=<container-app-name>
RG=<resource-group>
HOST=$(az containerapp show --name "$APP_NAME" --resource-group "$RG" \
  --query properties.configuration.ingress.fqdn -o tsv)
echo "https://$HOST"

Use this value as the base URL for all API calls below. You can also populate @baseUrl in test/test.cloud.http for quick testing. 将此值用作下方所有 API 调用的 base URL。你也可以填入 test/test.cloud.http 的 @baseUrl 进行快速测试。

Authentication 认证

All chat endpoints require a GitHub OAuth token. Pass it in both the Authorization and x-github-token headers: 所有聊天端点都需要 GitHub OAuth token。请在 Authorization 和 x-github-token 两个 header 中同时传入：

http 
Authorization: Bearer <gho_or_ghu_token>
x-github-token: <gho_or_ghu_token>
Tip: In secure environments, prefer per-user OAuth tokens (gho_ / ghu_) over shared service tokens to reduce key leakage risk. 提示：在安全要求较高的环境中，建议使用用户级 OAuth token（gho_ / ghu_），而非共享服务 token，以降低密钥泄露风险。

POST /agent/chat POST /agent/chat

Standard JSON request/response endpoint for agent chat. 标准 JSON 请求/响应的 agent 聊天端点。

Request 请求

bash 
curl -X POST "https://<host>/agent/chat" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <gho_or_ghu_token>" \
  -H "x-github-token: <gho_or_ghu_token>" \
  -d '{"prompt": "What is the price of a Standard_D4s_v5 VM in East US?"}'

Response 响应

json 
{
  "session_id": "abc123-def456-...",
  "response": "The agent's final response text",
  "response_intermediate": "Any intermediate responses",
  "tool_calls": ["list of tools invoked during the response"]
}

The response always includes a session_id, also returned via the x-ms-session-id response header. Use this ID for multi-turn conversations. 响应中始终包含 session_id，并且也会通过 x-ms-session-id 响应头返回。可以使用该 ID 进行多轮会话。

Multi-Turn Conversations 多轮会话

To resume an existing session, pass the session ID in the x-ms-session-id request header: 要恢复已有会话，在请求头中传入 x-ms-session-id：

bash 
# Follow-up — resumes the same session with full conversation history
curl -X POST "https://<host>/agent/chat" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer <gho_or_ghu_token>" \
  -H "x-github-token: <gho_or_ghu_token>" \
  -H "x-ms-session-id: abc123-def456-..." \
  -d '{"prompt": "If I run that VM 24/7 for a month, what would it cost?"}'

If you omit x-ms-session-id, a new session is created automatically and its ID is returned in the response. 如果省略 x-ms-session-id，系统会自动创建新会话，并在响应中返回该会话 ID。

POST /agent/chatstream (SSE) POST /agent/chatstream（SSE 流式）

Use POST /agent/chatstream to receive responses incrementally as Server-Sent Events. This is ideal for real-time UIs that need to display tokens as they arrive. 使用 POST /agent/chatstream 通过 SSE 增量接收响应。适合需要实时逐 token 显示的前端 UI。

Request 请求

bash 
curl -N -X POST "https://<host>/agent/chatstream" \
  -H "Content-Type: application/json" \
  -H "Accept: text/event-stream" \
  -H "Authorization: Bearer <gho_or_ghu_token>" \
  -H "x-github-token: <gho_or_ghu_token>" \
  -d '{"prompt": "Give me a quick summary of Azure Functions pricing."}'

To resume an existing session, pass x-ms-session-id the same way as /agent/chat. 如需恢复已有会话，可以像 /agent/chat 一样传入 x-ms-session-id。

SSE Event Types SSE 事件类型

Event Type事件类型 Description说明
session Contains the session_id for the conversation 包含当前会话的 session_id
delta Incremental text chunks as the agent generates output agent 生成输出时的增量文本块
intermediate Intermediate reasoning or response snippets 中间推理或响应片段
tool_start Emitted when the agent begins a tool invocation agent 开始调用工具时发出
tool_end Emitted when the tool invocation completes 工具调用结束时发出
message Final full response text 最终完整响应文本
done Stream completion signal 流结束信号

Example SSE Sequence SSE 事件序列示例

text 
data: {"type":"session","session_id":"..."}

data: {"type":"delta","content":"Hello"}

data: {"type":"tool_start","tool_name":"bash","tool_call_id":"..."}

data: {"type":"tool_end","tool_name":"bash","tool_call_id":"..."}

data: {"type":"message","content":"Hello...final"}

data: {"type":"done"}

MCP Server Endpoint MCP Server 端点

The deployed app exposes an MCP server endpoint for remote MCP clients: 已部署应用会暴露一个 MCP server 端点，供远程 MCP 客户端使用：

text 
https://<host>/runtime/webhooks/mcp

If your environment enables function keys for the MCP extension endpoint, pass the key in the x-functions-key header. 如果你的环境为 MCP extension 端点启用了 function key，需要通过 x-functions-key header 传入。

VS Code mcp.json Configuration VS Code mcp.json 配置

Use inputs with password: true so the MCP key is not hardcoded in the file: 使用 inputs 并设置 password: true，避免把 MCP key 硬编码进文件：

json 
{
  "inputs": [
    {
      "type": "promptString",
      "id": "functions-mcp-extension-system-key",
      "description": "Azure Functions MCP Extension System Key",
      "password": true
    },
    {
      "type": "promptString",
      "id": "functionapp-host",
      "description": "Container app host, e.g. fmaaca-xxxx.<env>.<region>.azurecontainerapps.io"
    }
  ],
  "servers": {
    "remote-mcp-function": {
      "type": "http",
      "url": "https://${input:functionapp-host}/runtime/webhooks/mcp",
      "headers": {
        "x-functions-key": "${input:functions-mcp-extension-system-key}"
      }
    }
  }
}

Chat UI (Root Route) 聊天 UI（根路由）

After deployment, open your app root URL to access the built-in single-page chat UI: 部署后，打开应用根 URL 即可访问内置单页聊天界面：

text 
https://<host>/

At first load, enter: 首次访问时需要输入：

  • Base URL — typically your ACA app URL Base URL — 通常是你的 ACA 应用 URL
  • GitHub OAuth token — a gho_ or ghu_ token GitHub OAuth token — gho_ 或 ghu_ token

The base URL is stored in browser local storage. The GitHub token is stored in session storage only (cleared when the browser session ends). Base URL 保存在浏览器 local storage 中；GitHub token 只保存在 session storage 中，浏览器会话结束后即清除。

URL Hash Prefill URL Hash 预填

You can prefill both values via URL hash to skip the initial setup screen: 你可以通过 URL hash 预填这两个值，跳过初始设置界面：

text 
https://<host>/#baseUrl=https%3A%2F%2F<host>&token=<url-encoded-token>

On load, the page reads these values, stores them in their respective storage locations, then removes the hash from the address bar. 页面加载后，会读取这些值，分别写入对应的 storage，并从地址栏中移除 hash。
